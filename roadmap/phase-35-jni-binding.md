# Phase 35 — JNI (JVM Desktop) Binding

**Status**: shipped 2026-05-16 · tag `phase-35-jni-binding` ·
release `0.87.0+abi.1.13.0`.

## Goal

Ship a desktop JNI binding for libp4a, gated by the shared
cross-binding parity fixture used by Octave / MATLAB / WASM /
R / Julia. The binding must be byte-equivalent to the native
Python reference. Android NDK packaging is a separate follow-up
phase (Phase 36) that reuses the same Java class and JNI source.

## Why JNI desktop first

* JDK toolchains are widely available and the parity gate runs in
  seconds — we can validate the JNI bridge without installing the
  ~3 GB Android NDK.
* The same `Pls4all.java` + `p4a_jni.c` files port to Android NDK
  with only a build-script change (use `$ANDROID_NDK/toolchains/...`
  instead of system gcc). Doing desktop first means Phase 36 is a
  packaging task, not a debugging task.
* JNI is the lingua franca of JVM-based environments: Spring,
  Quarkus, Kotlin Native, Scala, Clojure all consume it directly.

## Architecture

```
bindings/jni/
├── java/io/github/pls4all/Pls4all.java   Public Java API + JNI declarations
├── c/p4a_jni.c                            Implements
│                                          Java_io_github_pls4all_Pls4all_*
├── c/io_github_pls4all_Pls4all.h          Generated by javac -h (gitignored)
├── test/TestParity.java                   Parity gate vs shared fixture
├── build.sh                               One-shot build script
└── build/                                 Output (.so, .class) — gitignored
```

The Java class `io.github.pls4all.Pls4all`:
* `static String version()` — proxies `p4a_get_version_string`.
* `static int[] abiVersion()` — `{major, minor, patch}`.
* `static FitResult plsFit(double[] x, double[] y, int n, int p,
  int q, int nComponents)` — fits SIMPLS, returns a record-like
  `FitResult` (coefficients, x_mean, y_mean, predictions).

The native method routes through `p4a_pls_fit_simple` (ABI 1.13+
helper from Phase 33), using `GetPrimitiveArrayCritical` to pin the
Java buffers without copying. Java arrays are row-major, which is
also the libp4a row-major C ABI, so the pointer is passed straight
through — no transpose.

## Parity gate

```
$ LD_LIBRARY_PATH=build/dev-release/cpp/src \
  PLS4ALL_JNI_LIB=bindings/jni/build/libp4a_jni.so \
  $JAVA_HOME/bin/java -Drepo.root=$(pwd) \
    -cp bindings/jni/build/classes TestParity

Pls4all.version()        = 0.87.0+abi.1.13.0
fixture pls4all_version  = 0.87.0+abi.1.13.0
rmse_rel coefficients    = 0.000e+00
rmse_rel x_mean          = 0.000e+00
rmse_rel y_mean          = 0.000e+00
rmse_rel predictions     = 0.000e+00
PARITY GATE PASS
```

All four arrays match the fixture **bit-exactly** (rmse_rel = 0).
This is the strongest possible parity result — the JNI binding is
indistinguishable from the native C ABI.

## Implementation notes

* **GetPrimitiveArrayCritical**: the JNI spec forbids JNI callbacks,
  blocking, or throwing between Get and Release. We respect that:
  the native call is the only thing between the Get block and the
  Release block.
* **conda-forge openjdk layout**: the conda package places the JDK
  under `<env>/lib/jvm/` rather than at the env root. The build
  script auto-detects and adjusts.
* **System.load vs loadLibrary**: the static initializer prefers
  `PLS4ALL_JNI_LIB` (absolute path) when set, otherwise falls back
  to the standard `java.library.path` lookup. This keeps headless
  CI invocations explicit while letting packaged consumers rely on
  the JVM's library-path mechanism.

## ABI surface

No new symbols. Uses `p4a_pls_fit_simple` (ABI 1.13). ABI version
remains 1.13.0.

## Backlog

* **Phase 36 — Android NDK packaging**: reuse this source, swap
  the compiler to `$ANDROID_NDK/.../<abi>-linux-android-clang`,
  package into an AAR with `libp4a.so` + `libp4a_jni.so` for
  arm64-v8a and armeabi-v7a (plus x86_64 for emulator).
* `Pls4all.predict(modelHandle, x)` against a saved model — needs
  the prediction-handle helpers in the C ABI (already public for
  Python; the JNI surface just hasn't been wired up yet).
* Memory-pool reuse: the current FitResult allocates four fresh
  Java arrays per call. For high-throughput callers we can expose
  an overload that writes into caller-provided buffers.
