# Phase 32a — WASM binding: parity gate

**Status:** shipped — local tag `phase-32a-wasm-parity`
(`0.84.0+abi.1.12.0`).

Resolves the known issue from Phase 32: WASM PLS fit now matches the
native Python binding at numerical floor.

## Root cause

Emscripten 5.0.7 mis-compiles `p4a_model_fit` (and related entry
points) when invoked with `p4a_matrix_view_t*` arguments populated
from JavaScript. The same kernel called from a pure-C translation
unit inside the WASM module fits correctly — so the bug is in the
WASM code-gen for the matrix-view-pointer call path, not in the C++
numerics.

The bug was localised through a sequence of diagnostic variants:

1. `p4a_wasm_self_test` — everything in C. Works.
2. `p4a_wasm_self_test_copy_then_fit` — JS data, C-malloc copies. Works.
3. `p4a_wasm_self_test_data_ptrs` — JS data ptrs, C-stack views. Works.
4. `p4a_wasm_self_test_data_ptrs_int_p` — same with explicit `int p`. Works.
5. `p4a_wasm_self_test_with_buffers` — JS data + JS view ptrs. **Fails.**
6. `p4a_wasm_self_test_with_buffers_copy_view` — JS data + JS view ptrs +
   C-stack copy of view. **Still fails** (rules out struct-layout issues).
7. Forwarding `with_buffers_copy_view` to `data_ptrs_int_p` from inside
   the same WASM module restored correct results.

The conclusion: the function-signature shape `(view*, view*, ...)`
triggers an Emscripten optimiser misbehaviour. Functions with raw
double pointers + ints in their signature compile correctly.

## Workaround / production API

Two helpers added to `bindings/js/src/wasm_entry.c`:

- `p4a_wasm_pls_fit` — takes raw `(x, y, n, p, q, n_components,
  coefs_out, x_mean_out, y_mean_out, preds_out)` and runs SIMPLS PLS
  regression with default centring. Writes into caller-supplied
  output buffers.
- `p4a_wasm_pls_predict_from_coeffs` — pure prediction loop.

TypeScript `fitPls` / `predictPls` (and the legacy `Model` class)
funnel through these helpers; user code in JS never touches the
broken matrix-view-pointer path.

## Parity gate

`bindings/js/test/run_smoke.mjs` loads the WASM module, fits a
deterministic SIMPLS regression, and compares the result against a
fixture generated by `generate_parity_fixture.py` (which runs the
same fit through the native Python binding).
